## 1. Architectural Style

### 1.1 Primary Patterns

Rubric employs a **strict layered architecture** overlaid with a **pipeline processing model**:

```
CLI (cli.py)
  │
  ▼
Orchestrator (orchestrator.py)  ───  pipeline stage driver
  │
  ├──► Engine (engine/)          ───  state machine + scheduler + gates
  ├──► Agents (agents/)          ───  role-specific workers
  └──► Models (models/)          ───  domain types (shared vocabulary)
```

- **Layered**: Each layer depends only on layers below it. No upward or sideways imports.
- **Pipeline**: Story flows through 8 sequential stages (Inception → Planning → Design → Implementation → Review → Acceptance → Integration → Delivery → Done), each stage handled by specific agent roles.
- **State Machine**: `StoryState` with `VALID_TRANSITIONS` forms a deterministic finite automaton with backtracking (every non-terminal state can go to `BLOCKED`, and most can regress one step).
- **Event-Driven (embryonic)**: `WorkflowEvent` + `EventHandler` callbacks provide publish/subscribe hooks, though only the engine emits — no external subscribers exist in production code.

### 1.2 Layering Cleanliness

The layering is **exceptionally clean** for a codebase of this size:

| Layer | Internal Dependencies | Clean? |
|-------|----------------------|--------|
| `models/` | None | ✅ Pure domain |
| `context/` | None | ✅ Pure utility |
| `engine/` | `models/`, `context/` | ✅ |
| `agents/` | `models/`, `agents/base` | ✅ (with one exception — see §3.4) |
| `orchestrator.py` | `engine/`, `agents/`, `models/` | ✅ |
| `cli.py` | `orchestrator` only | ✅ |

The **only impurity** is `BaseAgent` importing `WorkflowEngine` (line 12 of `agents/base.py`). This creates a reverse knowledge flow: the agent layer knows about the engine layer above it. In a pure layered architecture, `BaseAgent` should depend only on `models/`. However, this import is used solely for the optional `bind()` / `produce_artifact()` convenience — it does not create a circular dependency (engine does not import agents).

### 1.3 Alignment with README

The README describes accurately:
- The pipeline stages and their sequence (ASCII diagram in README matches the 8 stages in `orchestrator.py`)
- The 7 agent roles with matching names (Alice PO through Frank DevOps)
- The TDD Red→Green→Refactor cycle
- The project structure

**Mismatches found:**
- The README's "Stories flow through a structured lifecycle — inception to delivery — driven by a team of AI agents" **implies** an autonomous engine loop. In reality, `orchestrator.run_full_pipeline()` manually drives each stage step-by-step, bypassing the engine's `run_story()` auto-loop entirely (line 77-258 of orchestrator.py vs line 244-266 of workflow.py). The autonomous `run_story()` exists but is **dead code** from the orchestrator's perspective.
- The README mentions "Quality gates at stage boundaries" via `TransitionGate`, but `orchestrator.py` never calls `validate_transition()` — it calls `story.transition()` directly. The gate system is **defined but unenforced**.

## 2. Design Pattern Usage

### 2.1 ABC / Template Method (`agents/base.py`)

**Pattern**: Abstract Base Class with `@abstractmethod execute(task, story)`.

**Implementation**: `BaseAgent` defines the interface; 7 concrete agents implement `execute()`. `DeveloperAgent` additionally adds `execute_step()` for finer-grained TDD control.

**Assessment**: ✅ **Appropriate.** The pattern is well-chosen. All agents follow the same contract. The optional `bind()` method (lines 29-32) provides a clean injection point for the engine dependency.

**Issue**: `DeveloperAgent` duplicates the TDD loop logic — it has both `execute_step()` (called by engine line 178) and `execute()` (line 52-86) which contains its own loop. The orchestrator calls `engine.execute_task_tdd()` which calls `execute_step()`, but if someone calls `DeveloperAgent.execute()` directly, they get a different codepath. Two ways to do the same thing breeds inconsistency.

### 2.2 State Machine (`models/story.py`)

**Pattern**: Explicit transition table via `VALID_TRANSITIONS: dict[StoryState, list[StoryState]]`.

**Implementation**: 10 states, each with an explicit list of valid target states. `Story.transition()` validates before applying (line 188-203). `advance_story()` in the engine navigates the state graph forward (workflow.py lines 220-240).

**Assessment**: ✅ **Solid.** The explicit transition table is readable, testable, and type-safe. The inclusion of `BLOCKED` as a universal escape and the ability to regress one step (`INCEPTION ← PLANNING ← DESIGN ← ...`) reflect real workflow needs.

**Issue**: The `advance_story()` method re-implements ordering logic (lines 228-234) by comparing indices in a hardcoded `stage_order` list against `VALID_TRANSITIONS`. This duplicates knowledge — the ordering is already implicit in the transition table. A simpler approach: iterate `stage_order` and pick the first `VALID_TRANSITIONS` entry that appears later.

### 2.3 Observer / Event (`engine/workflow.py`)

**Pattern**: `WorkflowEvent` objects + `EventHandler` callable type + `on_event()` registration + `_emit()` dispatch.

**Implementation**:
- `WorkflowEvent`: simple data class with `event_type`, `story_id`, `data` (lines 19-28)
- `EventHandler`: `Callable[[WorkflowEvent], None]` (line 31)
- Registration: `on_event(handler)` appends to a list (line 100-101)
- Dispatch: `_emit()` iterates handlers and calls each (lines 103-107)
- Events are also logged internally in `self._log` for audit

**Assessment**: ⚠️ **Minimal but functional.** The pattern is correctly implemented for its scope. However:
- No event type hierarchy — everything is a string `event_type`. This is fragile (no type checking, typos are silent).
- No async support — handlers are synchronous, which limits scalability.
- No event filtering or subscription scoping — all handlers get all events.
- Events are **never consumed externally** — the event log is only read via `status()`. This means the pattern exists but has no observable benefit in the current codebase.

### 2.4 Strategy (Transition Gates — `engine/transitions.py`)

**Pattern**: `TransitionGate` wraps a callable `check: Callable[[Story], bool]`. Gates are composed into per-stage lists in `DEFAULT_STAGE_GATES`. `validate_transition()` evaluates all gates for the current stage.

**Implementation**: 4 built-in gates (`ALL_TASKS_COMPLETE`, `HAS_ARTIFACTS`, `HAS_ACCEPTANCE_CRITERIA`, `MIN_PROGRESS`). The `check` callable makes each gate composable and testable.

**Assessment**: ✅ **Well-designed, but unused.** The pattern is elegant and extensible. However:
- `orchestrator.py` never calls `validate_transition()` — it calls `story.transition()` directly (e.g., line 77, 125). The gates are defined but **completely bypassed**.
- `MIN_PROGRESS` (requiring ≥50% progress) seems misaligned — it would block early-stage transitions that haven't started yet.
- The `BLOCKED` state has no associated gates, meaning blocked stories can transition to any prior state without checks.

**Critical gap**: The gate system is the intended quality enforcement mechanism, but the orchestrator ignores it. This is the architectural equivalent of building a lock but leaving the door open.

### 2.5 Scheduler / Dispatcher (`engine/scheduler.py`)

**Pattern**: `TaskScheduler.find_best_agent()` selects an agent based on role match + availability + utilization (load balancing).

**Implementation**: The ranking function (lines 55-57) produces a `(role_match, utilization)` tuple; Python's sort does the rest. Role match is binary (0 = matches, 1 = doesn't). Lower utilization is preferred.

**Assessment**: ✅ **Good for current scale.** Simple, readable, and effective for single-threaded use. The fallback to "any available agent" when no role match exists (line 47-48) is pragmatic.

**Gaps**:
- No capability-based matching — only role is considered. If two agents share a role, there's no way to prefer one based on skill match.
- No consideration of past agent affinity (e.g., the agent who designed the architecture might be best to implement it).
- No weighting — all roles are equal, all tasks are equal.

### 2.6 Factory (`BaseAgent.produce_artifact`, `create_default_team`)

**Pattern**: Two factory-like methods:
1. `BaseAgent.produce_artifact()` (agents/base.py lines 44-63) — creates and optionally registers `Artifact` objects.
2. `create_default_team()` (orchestrator.py lines 27-37) — instantiates all 7 agents.

**Assessment**: ✅ **Appropriate for the abstraction level.** `produce_artifact()` properly handles the optional engine binding (if agent is bound, artifact auto-registers). `create_default_team()` is a simple convenience factory.

### 2.7 Registry (`WorkflowEngine` registries)

**Pattern**: Three in-memory dictionaries on `WorkflowEngine`:
- `self.stories: dict[str, Story]` (line 49)
- `self.artifacts: dict[str, Artifact]` (line 50)
- `self.agents: dict[str, Agent]` (line 51)

Registration via `register_agent()`, `create_story()`, `add_artifact()`.

**Assessment**: ✅ **Simple and sufficient.** No over-engineering. The registries provide O(1) lookups. The convenience methods (`get_agents_by_role()`, `get_artifacts_for_story()`) add value without complexity.

**Gap**: No removal/deletion methods. Stories, agents, and artifacts accumulate indefinitely. For a long-running process, this is a memory leak.

## 3. Coupling Analysis

### 3.1 Overall Coupling Profile

| Module | Fan-out (internal) | Fan-in |
|--------|-------------------|--------|
| `models/` | 0 | 9 (highest) |
| `context/` | 0 | 1 |
| `engine/` | 5 (workflow), 2 (scheduler), 1 (transitions) | 2 (workflow) |
| `agents/` | 4 per agent | 0 |
| `orchestrator.py` | 5 | 1 |
| `cli.py` | 1 | 0 |

**Assessment**: Coupling is **appropriate for a 2,400-line codebase**. The hub-and-spoke pattern (models at center) is natural for domain-driven design. The pure modules (`models/`, `context/`) have zero internal deps — ideal.

### 3.2 When Coupling Feels Too Tight

1. **`models/story.py` is a gravity well (fan-in 9)**. Every single module except `context/` and `cli.py` imports it directly. A change to `Task` structure, `StoryState`, or `VALID_TRANSITIONS` ripples everywhere. For example:
   - Adding a new state requires: updating `VALID_TRANSITIONS`, `DEFAULT_STAGE_GATES`, `stage_order` in `advance_story()`, `STAGE_RESPONSIBILITIES`, and potentially every agent's `execute()`.
   - Changing `TaskStatus` values touches the engine, scheduler, all agents, and orchestrator.
   - This is **appropriate centralization** for the current scale, but is a risk vector as the codebase grows.

2. **`orchestrator.py` (fan-out 5)** — ties together engine, 3 model modules, and all 7 agent types. This is the "god module" that knows too much about too many things. Adding a new stage means modifying 264-line orchestrator with a new block that replicates the same pattern (find agent → create task → assign → execute → complete).

3. **`DeveloperAgent` has dual coupling** — it implements both `BaseAgent.execute()` (for generic use) and `execute_step()` (for engine's TDD loop). The engine calls `execute_step()`; the orchestrator calls `engine.execute_task_tdd()`. But `DeveloperAgent.execute()` contains its own TDD loop (lines 80-84) which would be used if someone calls `execute()` directly. This is **inconsistent interface coupling**.

### 3.3 When Coupling Feels Too Loose

1. **No protocol/interface for `ContextManager`** — the engine depends on the concrete class. If you wanted to swap in a Redis-backed context store, you'd need to change `WorkflowEngine.__init__()`. A `ContextProtocol` would be cleaner at scale, but is probably over-engineering for now.

2. **`TransitionGate.check` is `Any` typed** (transitions.py line 18). The constructor parameter says `Any` when it should be `Callable[[Story], bool]`. This loses type safety.

3. **No abstract `WorkflowEngine` interface** — `BaseAgent` imports the concrete `WorkflowEngine`, not a protocol. This is the primary coupling concern (see §3.4).

### 3.4 The BaseAgent → WorkflowEngine Import

```python
# agents/base.py, line 12
from rubric.engine.workflow import WorkflowEngine
```

**Is this a layering violation?** Technically **yes** — the agents layer imports from the engine layer above it. In a strict layered architecture, agents should only depend on `models/`.

**But is it harmful?** At current scale, **no**. The import is used only for:
1. The optional type annotation `self.engine: WorkflowEngine | None` (line 27)
2. The optional `bind(engine)` method (lines 29-32) which calls `engine.register_agent()`
3. The optional `produce_artifact()` shortcut (line 61-62) which calls `self.engine.add_artifact()`

All usages are **optional** — if `engine` is `None`, `produce_artifact()` still works (returns an unregistered `Artifact`). This is a convenience coupling, not a structural one.

**Fix**: Introduce a lightweight `EngineProtocol` (or `ArtifactRegistry` protocol) in `models/` that defines just `register_agent()` and `add_artifact()`. Have `WorkflowEngine` implement it. `BaseAgent` would import the protocol instead of the concrete engine. This inverts the dependency and eliminates the cross-layer import.

### 3.5 Summary

```
Coupling verdict: GOOD for current codebase size
          Risk: Low (the DAG is clean)
   Improvement: Add EngineProtocol in models/ to invert the BaseAgent→WorkflowEngine dependency
```

### 4.2 Problematic Cohesion

**`WorkflowEngine` (engine/workflow.py)** — 302 lines, 4 concerns:
1. **State machine transitions** (`transition_story`, `advance_story`, `check_stage_complete`)
2. **Task execution** (`assign_tasks_for_stage`, `execute_task_tdd`, `complete_task`)
3. **Event system** (`on_event`, `_emit`, `WorkflowEvent`, `EventHandler`)
4. **Registry management** (`register_agent`, `create_story`, `add_artifact`)

These are separable: the event system could be extracted into `engine/events.py`, registries into `engine/registry.py`. This would bring `WorkflowEngine` down to ~150 lines and give each class a single job.

**`orchestrator.py`** — 264 lines, monolithic. Each of the 8 stages repeats an identical pattern:
```python
# Lines 77-258, repeated 8 times with minor variations
engine.transition_story(story.id, StoryState.<STAGE>, "Entering <stage> stage")
agent = next(a for a in agents if isinstance(a, <AgentType>))
task = Task(title="...", required_role="...", ...)
story.tasks.append(task)
engine.scheduler.find_best_agent(...)
engine.scheduler.assign_task(...)
agent.execute(task, story)
engine.add_artifact(...)
engine.complete_task(story.id, task.id)
```

This is **copy-paste programming** at the module level. The pattern should be extracted into a method like `_run_stage(agents, story, target_state, agent_type, task_config)`.

**`DeveloperAgent`** — handles TDD steps (lines 38-50) AND non-TDD concerns (migrations lines 58-68, config lines 69-79) in the same `execute()` method. The migration/config paths feel tacked on and could be separate agents or handled via task type dispatch.

### 4.3 Would Split or Merge Improve Things?

| Candidate | Action | Rationale |
|-----------|--------|-----------|
| `WorkflowEngine` | **Split** — extract event system → `engine/events.py` | Clean separation, easier testing |
| `orchestrator.py` | **Refactor** — extract stage loop pattern → helper method | Eliminate 80% duplication |
| `DeveloperAgent` | **Keep** — but clarify dual interface | The execute/execute_step duality is confusing; pick one |
| `engine/workflow.py` + `scheduler.py` | **Keep separate** | Good separation of concerns |
| `models/story.py` | **Keep** | 231 lines is fine for a core model |

## 5. Extensibility

### 5.1 Adding a New Agent Role

**Effort**: Low (5 steps, ~30 minutes)

1. Add `Role` enum value in `models/agent.py` — **1 line**
2. Create agent class in `agents/` extending `BaseAgent` — **~60-150 lines**
3. Implement `execute(task, story)` — **core logic**
4. Export in `agents/__init__.py` — **1 line**
5. Add to `create_default_team()` and stage logic in `orchestrator.py` — **~15 lines**

**Assessment**: ✅ **Straightforward.** The `BaseAgent` contract is simple. The pattern is well-established across the 7 existing agents.

**Pain point**: The orchestrator's monolithic structure means adding an agent also means adding a new 10-15 line block to `orchestrator.run_full_pipeline()` that replicates the same pattern. There's no plugin registration where a new agent declares "I handle stage X" — the orchestrator must be edited manually.

### 5.2 Adding a New Pipeline Stage

**Effort**: Medium (requires changes in 4 files)

1. Add `StoryState` value — `models/story.py`, **1 line**
2. Add `VALID_TRANSITIONS` entries — `models/story.py`, **~2-4 lines**
3. Add `DEFAULT_STAGE_GATES` entry — `engine/transitions.py`, **~1-3 lines**
4. Add `STAGE_RESPONSIBILITIES` entry — `models/agent.py`, **~1 line**
5. Add stage block in `orchestrator.run_full_pipeline()` — **~10-15 lines**
6. Update `stage_order` in `advance_story()` if adding mid-sequence — `engine/workflow.py`, **1 line**
7. Optionally add new `ArtifactType` for stage outputs — `models/artifacts.py`, **1 line**

**Assessment**: ⚠️ **Fragile.** The same change must be propagated across 4-5 files manually. There's no single configuration that declares a stage and its properties. Missing any one file causes subtle bugs (e.g., a stage that can't be transitioned to, or has no responsible agents).

**Improvement**: A `PipelineConfig` that maps stages → responsible roles → required gates → expected artifacts would make this a single-entry-point change.

### 5.3 Adding a New Artifact Type

**Effort**: Low (1 line + agent code)

1. Add `ArtifactType` enum value in `models/artifacts.py` — **1 line**
2. Reference it in the producing agent's `execute()` — **inside existing agent code flow**

**Assessment**: ✅ **Very easy.** The `ArtifactType` enum is straightforward.

### 5.4 Integrating Real LLMs

**Effort**: Medium-High (architectural gap exists)

The `config/llm_config.json` provides a detailed schema (providers, models, per-agent temperature/system prompts). However:

1. **No runtime code reads it** — there is no `LLMClient`, no `LLMProvider` abstraction, no config loader. The JSON is a dead document.
2. **No `LLMProvider` protocol** — agents call `execute()` which returns template strings. To integrate an LLM, each agent's `execute()` method would need to be rewritten or an `LLMBackend` strategy injected.
3. **No streaming support** — the entire synchronous agent model assumes blocking `execute() → list[Artifact]`.
4. **No cost tracking** — no token counting, no budget management.

**Assessment**: ⚠️ **Prepared on paper, unprepared in code.** The config file shows intent but the architecture doesn't support clean LLM integration. An `LLMProvider` protocol (or injected into `BaseAgent.__init__`) would allow swapping providers per-agent.

### 5.5 Plugin / Extension Story

**Assessment**: ❌ **Not designed for extensions.**

- No plugin system, no hooks beyond the event system (which is unused).
- No dependency injection container — everything is manually wired in `orchestrator.run_full_pipeline()`.
- No way to register a custom agent without modifying orchestrator source.
- The `agents/__init__.py` serves as a manual registry but does not auto-discover.

For a v0.1, this is acceptable. For v1.0, a proper plugin mechanism (entry points, hook decorators, or a registry pattern) would be needed.

## 6. Testability

### 6.1 Can Models Be Tested in Isolation?

**Yes** — ✅ **Very high testability.**

`models/story.py`, `models/agent.py`, and `models/artifacts.py` are pure Pydantic models with zero rubric internal dependencies. They import only `pydantic`, `uuid`, `datetime`, and `enum`. All logic (state transitions, dependency checking, step progression) is pure computation with no I/O.

**Example tests that are trivial:**
- `test_valid_transition()` — transition from PLANNING to DESIGN succeeds
- `test_invalid_transition()` — transition from DONE to INCEPTION raises ValueError
- `test_task_ready_with_deps_met()` — `is_ready()` returns True when dependencies are done
- `test_task_not_ready()` — `is_ready()` returns False when dependencies are pending
- `test_story_progress()` — verify progress calculation with mixed step states

### 6.2 Can the Engine Be Tested Without Agents?

**Partially** — ⚠️ **Moderate testability.**

`WorkflowEngine` depends on:
- `TaskScheduler` (concrete class, instantiated in `__init__`, line 47)
- `ContextManager` (concrete class, instantiated in `__init__`, line 48)

The registries (stories, artifacts, agents) are plain `dict`s — easily populated for testing.

**Testing challenges:**
- `execute_task_tdd()` (line 149) requires a `agent_handler` argument — any object with an `execute_step()` method. This is testable via a mock/spy.
- `run_story()` is the full autonomous loop. It's testable but requires registering agents in the engine, which means agents must be imported. This creates a test dependency on `agents/`.
- There's no way to inject a mock `TaskScheduler` or `ContextManager` — both are hard-coded in `__init__`.
- The event system appends to `self._log` — testable via `status()["events_logged"]`, but no direct event assertion API exists.

**Improvement**: Make `scheduler` and `context` injectable via `__init__`:
```python
def __init__(self, scheduler=None, context=None):
    self.scheduler = scheduler or TaskScheduler()
    self.context = context or ContextManager()
```

### 6.3 Can Agents Be Tested Without the Engine?

**Yes** — ✅ **Very high testability.**

`BaseAgent.engine` is `None` by default (line 27). Agents can be instantiated and their `execute()` called with mock `Task` and `Story` objects. The `produce_artifact()` helper works without an engine — it simply doesn't auto-register (line 61: `if self.engine` guards the optional behavior).

**Pattern for pure agent tests:**
```python
def test_product_owner_creates_acceptance_criteria():
    agent = ProductOwnerAgent()
    task = Task(title="Define criteria", ...)
    story = Story(title="Login", ...)
    artifacts = agent.execute(task, story)
    assert len(artifacts) == 2
    assert artifacts[0].artifact_type == ArtifactType.STORY_BRIEF
```

**The only coupling**: If an agent tries to use `self.engine` without binding, it would get an `AttributeError` (accessing `.add_artifact()` on `None`). But `produce_artifact()` handles this with `if self.engine` guard. The risk is if a future agent adds a direct `self.engine.something` call.

### 6.4 Testability Improvement Summary

| Area | Current | Target |
|------|---------|--------|
| Models | ✅ Pure, no deps | No change needed |
| Engine | ⚠️ | Make `scheduler`/`context` injectable, expose event assertion API |
| Agents | ✅ Works without engine | No change needed |
| Orchestrator | ❌ Monolithic | Extract stage runner helper for targeted testing |
| CLI | ⚠️ | Thin wrapper — testing via `run_full_pipeline` mock is sufficient |

### 6.5 Dependency Injection Points

Current injection points:
- `BaseAgent.bind(engine)` — ✅ injects engine into agent
- `WorkflowEngine.on_event(handler)` — ✅ injects event handlers

**Missing injection points:**
- `WorkflowEngine.__init__()` — should accept optional `scheduler` and `context`
- `orchestrator.create_default_team()` — should accept optional agent overrides
- Agent config — should be injectable (e.g., LLM provider, capabilities)
- `TaskScheduler` — no strategy injection; hardcoded ranking algorithm

## 7. Scalability & Concurrency

### 7.1 Current Concurrency Model

**Single-threaded, synchronous, blocking.**

Everything runs on one thread. The orchestrator calls `agent.execute()` which blocks until all artifacts are returned. There is no `async`, no `threading`, no `asyncio`.

```
orchestrator.py          engine/workflow.py          agents/*.py
    │                        │                           │
    ├── sync call ──────────►│                           │
    │                        ├── sync call ─────────────►│
    │                        │◄── return artifacts ──────│
    │◄── return results ────│                           │
```

### 7.2 Could Multiple Stories Run Concurrently?

**Not without changes.** The barriers are:

1. **`WorkflowEngine` has shared mutable state** — `self.stories`, `self.artifacts`, `self.agents`, `self._log`, `self.event_handlers` are all plain `dict`/`list` with no locks. Concurrent writes would cause data corruption.

2. **`ContextManager` is not thread-safe** — `self._store` and `self._history` are plain `dict`/`list`. The `set()` method (manager.py lines 32-42) does a read-modify-write (`old = self._store.get(key)` then `self._store[key] = value`) — not atomic. Concurrent calls could lose updates or produce inconsistent history.

3. **`TaskScheduler` is stateless** — ✅ This is the one safe component. It receives agents as argument; it doesn't maintain internal state between calls.

4. **Agent objects have mutable state** — `Agent.active_tasks`, `completed_tasks` are mutable. Two threads assigning tasks to the same agent simultaneously would corrupt task tracking.

**Changes needed for concurrency:**
- Add `threading.Lock` or `asyncio.Lock` to `WorkflowEngine` for all registry mutations
- Add lock to `ContextManager.set()` for atomic read-modify-write
- Or: use a per-story granularity (one lock per story, not per engine)
- Replace blocking `agent.execute()` with async calls
- Or: adopt `concurrent.futures.ThreadPoolExecutor` for parallel stage execution

### 7.3 Thread Safety Assessment

| Component | Thread-Safe? | Risk |
|-----------|-------------|------|
| `WorkflowEngine` registries | ❌ No | Corrupted dict state under concurrent writes |
| `ContextManager._store` | ❌ No | Lost updates on concurrent `set()` |
| `ContextManager._history` | ❌ No | Race condition on append |
| `TaskScheduler` | ✅ Yes | Stateless — safe |
| `Agent` (Pydantic model) | ❌ No | Mutable `active_tasks` list |
| `Story` (Pydantic model) | ❌ No | Mutable `tasks`, `artifacts`, `history` lists |
| Event handlers | ❌ No | External handlers may have side effects |

### 7.4 Agent Capacity Limits — Are They Enforced?

**Partially.** The enforcement exists but is incomplete:

## 8. Error Handling & Resilience

### 8.1 Transition Error Handling

**Pattern**: `Story.transition()` raises `ValueError` for invalid transitions (story.py line 191-192). This is caught nowhere in the call chain — `orchestrator.py` calls `engine.transition_story()` → `Story.transition()` with no try/except.

```python
# orchestrator.py, line 77
engine.transition_story(story.id, StoryState.PLANNING, "Story created, entering planning")
# If this raises ValueError, the entire pipeline crashes.
```

**Assessment**: ❌ **Fragile.** Any invalid transition (e.g., logic bug in `advance_story()` that returns a wrong next state) causes an unhandled crash. The engine should catch transition errors, log them, and transition to `BLOCKED` instead.

### 8.2 Agent Execution Failure

**Pattern**: `agent.execute()` is called directly with no wrapper:

```python
# orchestrator.py, line 91
artifacts = po_agent.execute(inception_task, story)
```

**Assessment**: ❌ **No resilience.** If any agent's `execute()` raises an exception:
1. The pipeline crashes completely
2. No partial results are preserved
3. No retry is attempted
4. The story is left in an inconsistent state (task assigned but not completed)

**What should happen**:
- Wrap agent execution in try/except
- Log the error
- Optionally retry (configurable)
- If all retries fail, mark the task as `BLOCKED` and transition the story
- Continue with remaining tasks

### 8.3 Retry Logic

**Current**: **None.** There is no retry mechanism anywhere in the codebase.

The `llm_config.json` defines `max_retries: 3` and `retry_delay_seconds: 2` (lines 91-93), but these are never read or used. The config anticipates LLM calls that might fail transiently, but no runtime code implements the pattern.

### 8.4 Fallback Strategies

**Current**: **None.**

- If `find_best_agent()` returns `None` (no available agent), the orchestrator simply skips the task (line 89, 107, etc. — `if agent:` guard). The task remains unassigned, the story may never progress.
- There's no fallback to "try a different role" or "queue and retry later."
- `DeveloperAgent.execute_step()` is the only method with a fallback — and it just raises `ValueError` for unknown step types (line 50).

### 8.5 Stuck Detection in `run_story()`

**The `run_story()` auto-loop** (workflow.py lines 244-266) has basic stuck detection:

```python
if result is None and not story.ready_tasks() and story.tasks:
    logger.warning(f"Story {story_id} is stuck at {story.state.value}")
    break
```

**Assessment**: ⚠️ **Basic but functional.** It detects the case where `advance_story()` returns `None` (stage not complete because tasks remain) and no tasks are ready to execute (all blocked by dependencies or no available agents). However:
- It would **not** detect a story that's stuck with ready tasks but no available agents (because `ready_tasks()` would return tasks, keeping the loop running but never making progress).
- It does not detect **thrashing** — e.g., a loop where tasks complete but `advance_story()` fails, causing infinite iteration until `max_iterations` (200) is hit.
- The `max_iterations=200` default is arbitrary and could silently mask infinite loops on large stories.

## 11. Recommendations

### 11.1 Priority 1: Invoke the Quality Gates ⚡️ (High Impact, Low Effort)

**Problem**: `TransitionGate` system is defined but never called. The orchestrator calls `story.transition()` directly, bypassing all quality checks.

**Action**: Modify `WorkflowEngine.transition_story()` (workflow.py line 111) to call `validate_transition()` before allowing transitions. If gates fail, transition to `BLOCKED` instead of the requested state.

```python
def transition_story(self, story_id: str, new_state: StoryState, reason: str = "") -> None:
    story = self.get_story(story_id)
    ok, failures = validate_transition(story, new_state)
    if not ok:
        logger.error(f"Gate check failed: {'; '.join(failures)}")
        new_state = StoryState.BLOCKED
        reason = f"Gate failure: {'; '.join(failures)}"
    # ... proceed with transition
```

**Benefit**: Instantly activates the entire quality framework. All the work defining gates becomes observable. Stories that lack artifacts or acceptance criteria will correctly block.

### 11.2 Priority 2: Add Structured Error Handling ⚡️ (High Impact, Medium Effort)

**Problem**: Zero exception handling. Any failure crashes the pipeline.

**Action**: Wrap agent execution in try/except blocks with retry logic:

```python
def _safe_execute_agent(agent, task, story, max_retries=3):
    for attempt in range(max_retries):
        try:
            return agent.execute(task, story)
        except Exception as e:
            logger.warning(f"Agent {agent.name} failed attempt {attempt+1}: {e}")
            if attempt == max_retries - 1:
                task.status = TaskStatus.BLOCKED
                raise
            time.sleep(2 ** attempt)  # Exponential backoff
```

Apply the same pattern to:
- `orchestrator.py` — every agent execution block
- `WorkflowEngine.transition_story()` — catch invalid transitions
- `TaskScheduler.find_best_agent()` — catch agent interaction errors

**Benefit**: Pipeline survives transient failures. Partial progress is preserved. The system degrades gracefully by blocking individual tasks rather than crashing entirely.

### 11.3 Priority 3: Refactor Orchestrator to Eliminate Stage Duplication 🛠️ (Medium Impact, Medium Effort)

**Problem**: 8 nearly identical stage blocks in `run_full_pipeline()`, each 10-15 lines.

**Action**: Extract a helper method:

```python
def _run_stage(
    engine: WorkflowEngine,
    agents: list[BaseAgent],
    story: Story,
    target_state: StoryState,
    agent_type: type,
    task_config: dict,
) -> None:
    """Run a single pipeline stage: find agent → create task → execute → complete."""
    if story.state != target_state:
        engine.transition_story(story.id, target_state, f"Entering {target_state.value}")
    
    agent_obj = next(a for a in agents if isinstance(a, agent_type))
    task = Task(**task_config)
    story.tasks.append(task)
    
    agent = engine.scheduler.find_best_agent(task, list(engine.agents.values()), target_state.value)
    if agent:
        engine.scheduler.assign_task(task, agent)
        artifacts = agent_obj.execute(task, story)
        for art in artifacts:
            engine.add_artifact(art)
    engine.complete_task(story.id, task.id)
```

Then each stage becomes a single call:

```python
_run_stage(engine, agents, story, StoryState.REVIEW, ReviewerAgent, {
    "title": "Code review",
    "description": "Reviewer performs code review on all implementation",
    "required_role": "reviewer",
    "priority": TaskPriority.HIGH,
})
```

**Benefit**: ~200 lines of orchestrator becomes ~50. Adding a new stage is 1 call + stage config. Testing the stage runner becomes possible.
