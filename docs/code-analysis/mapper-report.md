## 6. Key Observations

### 6.1 Central Hub Module
**`rubric.models`** (especially `story.py`, `agent.py`, `artifacts.py`) is the undisputed hub. Every module except `context/` and `cli.py` directly imports model types. It has the highest fan‑in (6+ at package level, 9 at file level). Any change to model schemas ripples through the entire system.

### 6.2 Pure Modules (Zero Internal Dependencies)
- **`rubric.models`** — all sub‑modules (`story.py`, `agent.py`, `artifacts.py`)
- **`rubric.context.manager`** — no rubric imports whatsoever

These modules are the most reusable, testable, and isolated parts of the system.

### 6.3 Layering Cleanliness
The architecture follows a **strict layered pattern**:

```
CLI → Orchestrator → Engine → Models
                   → Agents → Models, Context
```

- **No upward dependencies** — lower layers never import from upper layers.
- **No sideways dependencies** — `engine/` does not import `agents/`; `agents/` does not import `engine/` (except for `BaseAgent` which imports `WorkflowEngine` — this is a **minor exception** that creates a cross‑layer link).
- **No circular dependencies** — the graph is a pure DAG.
- **Single entry point** — `cli.py` only touches `orchestrator`, preserving encapsulation.

### 6.4 Notable Design Details
1. **ContextManager sits alone** — it has fan‑out 0 and is only consumed by `engine/workflow.py`. This makes it a narrow, focused utility (likely thread‑safe state holder).
2. **BaseAgent imports WorkflowEngine** — this means every concrete agent transitively depends on the engine layer. If the goal is to keep agents engine‑agnostic, a dependency inversion (e.g., an `EngineProtocol` in models) could be used.
3. **`engine/transitions.py` is the leanest engine file** — fan‑out of 1 (only `models.story`), no agent or context dependency. It is a pure state‑machine validator.
4. **Orchestrator has the highest fan‑out (5)** — it ties together engine, models, and all agents. This is the natural "glue" module and carries the most integration risk.
5. **No abstract interfaces** — all dependencies are concrete imports. Introducing abstract base classes or protocols for `ContextManager` and `TaskScheduler` would further reduce coupling, though it may be over‑engineering for this codebase size.
